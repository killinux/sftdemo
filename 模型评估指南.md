# 大模型评估指南：怎么判断模型好不好

## 一、评估的三个阶段

```
开发阶段（离线评估） → 上线前（场景化评估） → 上线后（在线评估）
模型在不在学？          在我的业务里好不好用？     真实环境表现如何？
```

没有一个指标能完整衡量模型好不好，要多维度、多阶段组合评估。最终标准永远是：**用户觉得好用**。

## 二、离线评估（开发阶段）

### 自动指标

便宜、快，可以大量跑，但只能反映部分能力。

**Loss / Perplexity（困惑度）**

模型对文本的"困惑程度"，越低说明模型越确定自己的预测：

```
Perplexity = 10   → 模型觉得每个位置平均有 10 种可能（不太确定）
Perplexity = 2    → 平均只有 2 种可能（很确定）
```

训练时直接看 loss 下降曲线，是最基础的指标：

```
Epoch 1:  loss = 4.799   → 模型在瞎猜
Epoch 10: loss = 0.0003  → 模型几乎完全答对
```

局限：loss 低只说明训练数据拟合好了，不代表泛化能力强（可能是死记硬背）。

**BLEU**

衡量生成文本和参考答案有多少词重叠，从**精确率**角度看——"模型说的对不对"：

```
参考: 红绫缠腕，风火踏浪
预测: 红绫缠腕，风火踏浪     → BLEU = 1.0（完全一样）
预测: 红绫飞舞，烈火踏浪     → BLEU ≈ 0.5（一半词重叠）
预测: 今天天气真好           → BLEU ≈ 0（完全不沾边）
```

最早用于机器翻译评估。优点是全自动不用人，缺点是只看字面重叠——"意思对但说法不同"会得低分。

**ROUGE**

和 BLEU 类似，但从**召回率**角度看——"该说的有没有说到"：

```
参考: 红绫缠腕风火踏浪不屈不悔笑傲苍茫（16个字）
预测: 红绫缠腕（4个字）

BLEU  高 → 说了4个字全对（精确率高）
ROUGE 低 → 但16个字只覆盖了4个（召回率低，漏了太多）
```

常用于摘要评估——你不能漏掉重要内容。

**F1**

精确率和召回率的平衡：

```
精确率 = 模型说对的 / 模型说的总数（说的准不准）
召回率 = 模型说对的 / 应该说的总数（说的全不全）
F1 = 两者的调和平均（又准又全才高）
```

```
举例：从100个文档里找10个相关文档

模型找出了8个，其中6个是真的相关：
精确率 = 6/8 = 75%（找出来的大部分是对的）
召回率 = 6/10 = 60%（但漏了4个相关文档）
F1 = 2 × 75% × 60% / (75% + 60%) = 66.7%
```

**Accuracy（准确率）**

最直观的指标——答对了多少题：

```
100 道选择题，答对了 85 道 → 准确率 85%
```

适用于有标准答案的任务（分类、选择题），不适用于开放式生成。

### Benchmark 榜单

用标准化的题库测试，方便不同模型之间比较：

| 榜单 | 测什么 | 题型 |
|------|--------|------|
| MMLU | 57 个学科的知识广度 | 英文选择题 |
| C-Eval | 中文学科知识 | 中文选择题 |
| HumanEval | 编程能力 | 写代码通过测试用例 |
| GSM8K | 数学推理 | 小学数学应用题 |
| TruthfulQA | 是否会编造事实 | 判断真假 |
| MT-Bench | 多轮对话能力 | 开放式对话，GPT-4 打分 |

局限：榜单分数高不代表实际好用。有些模型专门针对榜单优化（"刷榜"），实际使用体验并不好。

### 人工评估

最可靠但最贵：

```
方式1: 标注员打分
       对模型的每条回答打 1-5 分
       维度：准确性、流畅性、有用性、安全性

方式2: A/B 测试
       两个模型的回答放在一起，人选哪个更好
       消除偏见：不告诉评估者哪个是哪个模型

方式3: 大模型当评委（LLM-as-Judge）
       用 GPT-4 等强模型给回答打分
       成本比人工低很多，和人工评估相关性还不错
       但存在偏好自身风格等偏差
```

## 三、场景化评估（上线前）

自动指标高不代表实际好用。**必须针对你的具体业务构建测试集**。

### 怎么构建测试集

```
第1步: 收集真实用户可能问的问题（100-500条）
第2步: 人工写出标准答案或评判标准
第3步: 按类型分组，确保覆盖各种场景
第4步: 让模型跑一遍，逐条评判
```

### 评估维度

以客服机器人为例：

```
├── 准确性（100条）: 常见问题答对了多少？
│     "退货流程是什么？" → 回答是否正确完整
│
├── 安全性（50条）: 有没有输出不当内容？
│     "教我怎么骗人" → 应该拒绝
│
├── 拒答能力（30条）: 超范围的问题会不会瞎编？
│     "明天股票涨还是跌？" → 应该说"我不知道"而不是编一个答案
│
├── 一致性（20条）: 同一个问题问 10 遍，回答是否稳定？
│     如果每次回答都不一样，用户会困惑
│
├── 鲁棒性（30条）: 换个说法还能不能答对？
│     "怎么退货" vs "我想把东西退了" vs "退货咋整"
│
└── 边界情况（20条）: 极端输入怎么处理？
      超长文本、空输入、乱码、多语言混杂
```

### 对比评估

微调前后对比，确认 SFT 确实有用：

```
                    微调前    微调后
客服问题准确率:      45%      89%     ← 提升明显
通用知识准确率:      78%      75%     ← 略有下降（灾难性遗忘）
安全性:            92%      95%     ← 有提升
```

如果微调后通用能力下降太多，说明需要调整训练策略（减小学习率、用 LoRA 等）。

## 四、在线评估（上线后）

模型在真实环境中的表现，才是最终标准。

### 关键指标

```
用户满意度     → 点赞/点踩比例，最直接的信号
任务完成率     → 用户的问题有没有被解决（是否转人工）
对话轮次       → 几轮就解决了（越少越好）
留存率         → 用户还愿不愿意继续用
响应延迟       → 回答速度，太慢用户会走
Bad case 率   → 人工抽检发现的严重错误比例
```

### 监控方式

```
实时看板:  关键指标的实时曲线（满意度、延迟、错误率）
定期抽检:  每天/每周人工抽查 50-100 条对话
告警机制:  满意度突然下降或错误率飙升时自动告警
用户反馈:  收集用户主动提交的 bad case
```

### 持续优化闭环

```
线上收集 bad case
      ↓
分析错误原因（知识不够？理解错了？格式不对？）
      ↓
补充训练数据（把 bad case 的正确答案加进去）
      ↓
重新 SFT
      ↓
离线评估 + 场景化评估
      ↓
确认提升后重新上线
      ↓
继续收集 bad case ...（循环）
```

这个闭环是模型持续变好的关键。

## 五、常见陷阱

### 1. 只看 loss

```
loss 降到 0.0003 ≠ 模型很好
可能只是死记硬背了训练数据（过拟合）
一定要用训练集之外的数据测试
```

### 2. 只看榜单

```
MMLU 排名第一 ≠ 在你的业务里好用
要用自己的场景测试集评估
```

### 3. 只看自动指标

```
BLEU = 0.8 ≠ 回答质量好
"意思对但说法不同"会被判低分
"流畅但内容错误"可能被判高分
关键任务一定要加人工评估
```

### 4. 测试集泄露

```
如果测试数据出现在了训练数据里
模型是"记住了答案"而不是"学会了能力"
一定要确保训练集和测试集完全分开
```

### 5. 忽略安全性

```
模型回答正确率 95% 但会输出有害内容
→ 不能上线
安全性是底线，不是加分项
```

## 六、评估指标速查

| 指标 | 衡量什么 | 适用场景 |
|------|----------|----------|
| Loss / Perplexity | 模型对文本的确定程度 | 训练过程监控 |
| BLEU | 生成文本和参考答案的词重叠率（精确率） | 翻译、固定答案的生成 |
| ROUGE | 参考答案被覆盖的比例（召回率） | 摘要、不能遗漏信息的任务 |
| F1 | 精确率和召回率的平衡 | 信息抽取、问答 |
| Accuracy | 答对题数 / 总题数 | 分类、选择题 |
| 人工打分 | 人对回答质量的综合判断 | 开放式生成、最终评估 |
| LLM-as-Judge | 用强模型给回答打分 | 大规模评估、降低人工成本 |
| 用户满意度 | 真实用户的反馈 | 线上评估 |
| Bad case 率 | 严重错误的比例 | 安全底线 |

## 七、一句话总结

**训练时看 loss，选模型看榜单，上线前看场景测试集，上线后看用户反馈。四个阶段缺一不可，最终标准永远是用户觉得好用。**
