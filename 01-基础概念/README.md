# 基础概念：从零理解大模型与 SFT

> 这是 SFT 教程的第一章，面向完全零基础的读者。读完这一章，你会理解大模型是什么、SFT 在干什么、以及训练过程中那些术语到底是什么意思。

---

## 一、什么是大模型

### 大模型 = 一大堆数字

大语言模型（LLM，Large Language Model）说白了就是**一大堆数字**，这些数字叫做"参数"。

比如 Qwen2.5-0.5B 这个模型，有 **5 亿个参数**（0.5B = 0.5 Billion = 5亿）。你可以把它想象成一个有 5 亿个旋钮的巨大机器，每个旋钮都调到了一个特定的位置，这些位置组合在一起，就让机器学会了"说话"。

常见模型的参数量：

| 模型 | 参数量 | 类比 |
|------|--------|------|
| Qwen2.5-0.5B | 5 亿 | 一本厚字典 |
| Qwen2.5-7B | 70 亿 | 一个小型图书馆 |
| Qwen2.5-72B | 720 亿 | 一个大型图书馆 |
| GPT-4（传闻） | 万亿级 | 一座城市的所有书 |

### 大模型怎么工作

大模型的工作方式非常简单——**一个字一个字地预测下一个字**：

```
你输入: "今天天气"
模型想: 下一个字最可能是"真" → "真"
模型想: 下一个字最可能是"好" → "好"
模型想: 下一个字最可能是"啊" → "啊"

最终输出: "今天天气真好啊"
```

没错，大模型就是一个**超级补全工具**。它不是"理解"了你说的话然后"思考"出答案，而是根据前面的字，预测后面最可能出现的字。只不过它预测得太准了，看起来就像真的在思考一样。

### 模型内部结构

模型内部长这样（以 Qwen2.5-0.5B 为例）：

```
"请你给哪吒写一首诗"
      ↓
┌─────────────────────────────────┐
│  Embedding 层（1.36亿参数）      │  ← 第一步：查字典，把每个字变成一串数字
└─────────────────────────────────┘
      ↓
┌─────────────────────────────────┐
│  Transformer 第 1 层            │  ← 浅层：认字、识别词性
│  Transformer 第 2 层            │
│  ...                            │
│  Transformer 第 12 层           │  ← 中层：理解句子含义
│  ...                            │
│  Transformer 第 24 层           │  ← 深层：综合理解，构思答案
└─────────────────────────────────┘
      ↓
┌─────────────────────────────────┐
│  输出层                          │  ← 最后一步：预测下一个字
└─────────────────────────────────┘
      ↓
"红绫缠腕，风火踏浪..."
```

三个部分各司其职：

- **Embedding 层** = 查字典。把文字变成数字向量，每个字互不影响，就像查字典一样——查"猫"的时候不需要知道旁边是什么字。
- **Transformer 层** = 读懂句子。让字和字之间互相看，理解上下文。比如"苹果很好吃"和"苹果发布新手机"里的"苹果"意思完全不同，Transformer 能通过上下文区分。24 层结构一样，但各自学到不同深度的理解。
- **输出层** = 预测下一个字。把 Transformer 的理解结果转化成"每个字出现的概率"，概率最高的就是预测结果。

### Decoder-only 架构

现在主流的大模型（GPT、Qwen、LLaMA、DeepSeek 等）都是 **Decoder-only** 架构，也叫"生成型"架构。

核心规则：**每个字只能看到前面的字，不能偷看后面的字**。

```
输入: "今 天 天 气 真 好"

"今" 能看到: [今]
"天" 能看到: [今, 天]
"天" 能看到: [今, 天, 天]
"气" 能看到: [今, 天, 天, 气]
"真" 能看到: [今, 天, 天, 气, 真]
"好" 能看到: [今, 天, 天, 气, 真, 好]
```

为什么不能看后面？因为后面的字还没生出来呢！模型是一个字一个字往后生成的，生成"真"的时候，"好"还不存在。

## 二、什么是 SFT

### SFT = Supervised Fine-Tuning = 有监督微调

大模型的训练分两个阶段：

```
第一阶段 —— 预训练：
  读了几万亿字的书（维基百科、网页、代码、论文...）
  → 学会了语言能力，能说流利的话
  → 但不听指令，你问它问题，它可能接着你的话继续往下编

第二阶段 —— SFT：
  用标注好的"问题 → 回答"数据再训练一下
  → 学会了按指令回答问题
  → 变成了一个"有问必答"的助手
```

### 类比：上学 vs 上班

```
预训练 = 上学：
  读了十几年书，学了语文数学英语物理化学...
  知识面很广，但从来没工作过
  你让他写个报告，他可能给你写篇散文

SFT = 入职培训：
  老板说："客户问价格，你这样回答..."
  "客户投诉，你这样处理..."
  培训几天后，就知道怎么按规矩干活了
```

**一句话总结：预训练给了模型语言能力，SFT 教它听话。**

### 为什么需要 SFT

没经过 SFT 的模型（也叫"基座模型"）是这样的：

```
你: 请介绍一下北京
基座模型: 请介绍一下上海。请介绍一下广州。请介绍一下深圳...
         （它把你的话当成了文本的一部分，继续往下补全）

你: 1+1等于几？
基座模型: 2+2等于几？3+3等于几？
         （它在做数学题的补全，而不是回答你的问题）
```

经过 SFT 之后：

```
你: 请介绍一下北京
SFT模型: 北京是中国的首都，位于华北平原...（正经回答）

你: 1+1等于几？
SFT模型: 1+1等于2。（直接回答）
```

## 三、SFT 核心原理

### SFT 和预训练有什么区别

代码流程几乎一模一样，**唯一的区别是：SFT 只对回答部分计算 loss，不对指令部分计算 loss**。

### loss 是什么

模型预测下一个字的时候，会给出一个概率分布——"我觉得下一个字是A的概率 30%，是B的概率 5%，是C的概率 60%..."

loss 就是**模型预测和正确答案之间的差距**：

```
正确答案是 "恩"

训练前:
  模型预测: "恩" 的概率 5%, "的" 的概率 20%, "是" 的概率 15%...
  → loss 大（猜错了，离正确答案很远）

训练后:
  模型预测: "恩" 的概率 95%, "的" 的概率 1%, "是" 的概率 0.5%...
  → loss 小（猜对了，非常接近正确答案）
```

**loss 大 = 猜错了，loss 小 = 猜对了。训练的全部目的就是让 loss 变小。**

类比：考试成绩。loss 就像"错误率"——第一次考试错 80%，复习后错 5%，说明学会了。

### 怎么做到"只对回答算 loss"

关键操作：**把指令部分的 label 设为 -100**。

PyTorch 的 CrossEntropyLoss 有一个特殊约定：遇到 label 值为 -100 的位置，就自动跳过，不计算 loss。

```
完整输入序列:

  请你给敖丙写一首诗：碧海生龙子，云中舞雪霜。恩仇难两忘，何处是家乡？<eos>
  |______________ 指令 + 上文 _____________||________ 回答 ________|

labels 设置:

  -100  -100  -100  -100 ... -100  -100   恩仇难两忘...    <eos>
  |__________ 不算 loss，跳过 __________|   |____ 只算这部分 ____|
```

为什么要这样做？

- 指令部分是人写的，不需要模型去"学习预测"指令内容
- 模型只需要学"看到这个指令后，应该回答什么"
- 如果连指令也算 loss，模型会花精力去"背题目"，浪费学习能力

### 类比

```
老师出题: "1 + 1 = ?"

普通训练（预训练）:
  让学生学会预测整个序列 → 既要学背题目 "1+1=?" 又要学答案 "2"

SFT:
  题目部分 label=-100，不用学 → 学生只需要记住答案 "2"

这样学生的全部精力都花在"学怎么回答"上，效率更高。
```

## 四、Attention 机制

Attention（注意力机制）是 Transformer 的核心，也是大模型能"理解上下文"的关键。

### 为什么需要 Attention

没有 Attention 的话，每个字只能看到自己：

```
没有 Attention:
  "苹果" → 只知道"苹"是"苹"，"果"是"果"
  不知道这是水果还是手机品牌

有了 Attention:
  "苹果很好吃" → "苹果"看到了"好吃" → 哦，这是水果
  "苹果发布新手机" → "苹果"看到了"手机" → 哦，这是品牌
```

### Attention 在做什么

简单说：**每个字"看"所有其他字，判断谁对自己最重要**。

```
句子: "小明 喜欢 吃 苹果"

当模型处理"吃"这个字时:
  看看"小明" → 重要性 0.1（知道谁在吃，有点用）
  看看"喜欢" → 重要性 0.3（"喜欢吃"是个搭配）
  看看"吃"   → 重要性 0.2（自己看自己）
  看看"苹果" → 重要性 0.4（吃什么？苹果！最重要）
```

这些"重要性分数"就是 **Attention Score（注意力分数）**。分数越高，说明这个字对当前字越重要。

### Query、Key、Value：图书馆类比

Attention 的计算涉及三个概念：Query（查询）、Key（键）、Value（值）。

类比：**去图书馆找书**

```
你要查的问题（Query）:  "我想了解机器学习"
书架上的书名（Key）:    "机器学习入门"、"Python编程"、"深度学习实战"、"做菜大全"
书的内容（Value）:      每本书的实际内容

查找过程:
  1. 拿你的问题（Query）去和每本书的书名（Key）对比
  2. 算出相关度: "机器学习入门" 0.5, "深度学习实战" 0.3, "Python编程" 0.15, "做菜大全" 0.05
  3. 按相关度加权，把对应的书的内容（Value）混合起来
  4. 相关度高的书内容占比大，不相关的书几乎忽略
```

在 Transformer 里：

```
每个字都会生成三个向量:
  Q（Query）= 我在找什么信息？
  K（Key）  = 我有什么信息可以提供？
  V（Value）= 我的具体内容是什么？

计算过程:
  1. 当前字的 Q 和所有字的 K 做点积 → 得到注意力分数
  2. 分数做 softmax → 归一化成概率
  3. 用概率对所有字的 V 加权求和 → 得到这个字的新表示
```

### Multi-Head Attention：多角度看句子

一个 Attention Head 只从一个角度看句子。但理解语言需要多个角度：

```
句子: "她 在 银行 存 了 钱"

Head 1（语法角度）: "她" → "存" → 关注"谁做了什么"
Head 2（语义角度）: "银行" → "钱" → 关注"在哪里做了什么"
Head 3（搭配角度）: "存" → "钱" → 关注"动词和宾语搭配"
Head 4（上下文角度）: "银行" → "存" + "钱" → 确认是金融机构不是河岸
```

**Multi-Head Attention = 多个注意力头同时工作，每个头关注不同的语言特征，最后把结果合并。**

就像一群人看同一部电影，有人关注剧情，有人关注画面，有人关注音乐，合在一起就是对电影的全面理解。

Qwen2.5-0.5B 有 14 个注意力头，意味着每一层都从 14 个不同角度分析句子。

### Attention Score 示意图

```
        小明  喜欢   吃  苹果
小明   [ 0.6  0.1  0.1  0.2 ]   ← "小明"最关注自己
喜欢   [ 0.3  0.4  0.2  0.1 ]   ← "喜欢"关注"小明"（谁喜欢）
 吃    [ 0.1  0.3  0.2  0.4 ]   ← "吃"最关注"苹果"（吃什么）
苹果   [ 0.1  0.1  0.4  0.4 ]   ← "苹果"关注"吃"和自己

每一行的分数加起来 = 1（经过 softmax 归一化）
分数越高，表示这两个字的关联越强
```

### 为什么 Decoder 用 Masked Attention

前面说过，Decoder 架构中每个字只能看到前面的字。怎么实现的？用 **Mask（遮罩）**：

```
        小明  喜欢   吃  苹果
小明   [ 0.9  -inf -inf -inf ]   ← 只能看自己
喜欢   [ 0.4  0.6  -inf -inf ]   ← 只能看前两个
 吃    [ 0.1  0.3  0.6  -inf ]   ← 只能看前三个
苹果   [ 0.1  0.1  0.3  0.5  ]   ← 能看所有（因为它是最后一个）

-inf 表示"完全看不到"，softmax 之后变成 0
```

这就叫 **Causal Mask（因果遮罩）**——不让模型偷看未来的信息。因为在实际生成的时候，未来的字还没生成出来，训练时也不能让模型偷看。

## 五、Tokenizer（分词器）

### 模型不认字，只认 Token

大模型不是按"字"或"词"处理文本的，而是按 **Token** 处理。Token 是模型处理文本的最小单位。

Tokenizer（分词器）就是把文本切成 Token 的工具。

### 怎么切的

不同语言、不同词汇的切法不一样：

```
中文:
  "哪吒降世" → ["哪吒", "降", "世"]          3 个 token
  "人工智能" → ["人工", "智能"]              2 个 token

英文:
  "Hello"    → ["Hello"]                    1 个 token
  "unhappiness" → ["un", "happi", "ness"]   3 个 token

混合:
  "GPT是AI" → ["GPT", "是", "AI"]           3 个 token

代码:
  "print('hello')" → ["print", "('", "hello", "')"]  4 个 token
```

可以看出：
- 常见的词通常是 1 个 token（"Hello"、"人工"）
- 罕见的词会被拆成多个 token（"unhappiness" → 3 个）
- 中文通常 1-2 个字为一个 token

### 为什么不同模型的 Tokenizer 不能混用

每个模型在训练时都有自己的"词表"（Vocabulary），就像每个模型有自己的字典：

```
Qwen 的词表:
  0 = "<pad>"
  1 = "<eos>"
  ...
  2345 = "人工"
  2346 = "智能"

LLaMA 的词表:
  0 = "<s>"
  1 = "</s>"
  ...
  2345 = "the"        ← 同一个编号，对应的字完全不同！
  2346 = "and"
```

如果用 LLaMA 的 Tokenizer 处理文本，再送给 Qwen 模型，模型拿到编号 2345，以为是"人工"，其实是"the"——结果就是乱码。

**所以：模型和 Tokenizer 必须配套使用，绝不能混搭。**

### 词表大小的影响

```
词表小（32000 个 token）:
  常见词够用，罕见词要拆成多个 token
  "清华大学" → ["清", "华", "大", "学"]（4个token，处理慢）

词表大（150000 个 token）:
  "清华大学"作为一个整体在词表里
  "清华大学" → ["清华大学"]（1个token，处理快）
  但模型的 Embedding 层参数更多，模型更大
```

词表大小是一个权衡：太小则表达能力不足，太大则模型参数多、训练慢。

### 特殊 Token

除了普通文字对应的 token，还有一些特殊 token，用来传递控制信号：

| Token | 全称 | 作用 | 类比 |
|-------|------|------|------|
| BOS | Beginning of Sequence | 表示序列开始 | 文章的标题 |
| EOS | End of Sequence | 表示序列结束，模型看到就停止生成 | 文章结尾的"完" |
| PAD | Padding | 填充短序列，让一个 batch 内长度对齐 | 考试试卷的空白部分 |
| UNK | Unknown | 词表里没有的字用这个代替 | 字典里查不到的生僻字 |
| SEP | Separator | 分隔不同段落 | 章节分隔线 |

实际使用中的例子：

```
原始文本: "你好，世界"

加上特殊 token 后:
  <BOS> 你好，世界 <EOS> <PAD> <PAD> <PAD>
  |开始|   正文    |结束|    填充到固定长度   |
```

在对话场景中，Chat Template 会用更复杂的特殊 token 来区分角色：

```
<|im_start|>user
你好<|im_end|>
<|im_start|>assistant
你好！有什么可以帮你的吗？<|im_end|>
```

这些特殊 token 让模型知道"谁在说话"、"什么时候该停"。

## 六、关键概念速查

| 概念 | 英文 | 一句话解释 | 类比 |
|------|------|-----------|------|
| 预训练 | Pre-training | 读几万亿字的书，获得语言能力 | 上学十几年 |
| SFT | Supervised Fine-Tuning | 用标注数据微调，教模型听指令 | 入职培训 |
| Embedding | Embedding | 把文字变成数字向量，每个字查表得到 | 查字典 |
| Transformer | Transformer | 让字和字互相看，理解上下文关系 | 全班同学互相讨论 |
| Decoder | Decoder | 只能看前文不能看后文的 Transformer | 考试时不能看后面的题 |
| Encoder | Encoder | 能看全文的 Transformer，用于理解任务 | 先把整篇文章读完再答题 |
| Loss | Loss | 模型预测和正确答案的差距，越小越好 | 考试错误率 |
| Epoch | Epoch | 所有训练数据从头到尾过一遍 | 刷一遍题库 |
| Label=-100 | Label Masking | 告诉 PyTorch 这个位置不用学，跳过 | 老师说"这题不考" |
| 参数 | Parameters | 模型里的数字，训练就是调这些数字 | 机器上的旋钮 |
| Tokenizer | Tokenizer | 分词器，把文字切成 token | 把句子拆成一个个卡片 |
| 梯度 | Gradient | 告诉每个参数该调大还是调小 | 导航里的"往左转/往右转" |
| 反向传播 | Backpropagation | 从 loss 反推每个参数的梯度 | 考完试对答案，知道哪里错了 |
| 学习率 | Learning Rate | 参数每次调整的幅度 | 每步走多远 |

### 补充说明

**Decoder vs Encoder：**

```
Encoder（编码器）:
  每个字能看到所有字（前面和后面都能看）
  适合"理解"任务：文本分类、情感分析
  代表模型：BERT

Decoder（解码器）:
  每个字只能看到前面的字
  适合"生成"任务：写文章、对话、写代码
  代表模型：GPT、Qwen、LLaMA

现在主流的大模型几乎都是 Decoder-only 架构。
```

**Epoch 的直觉理解：**

```
Epoch 1:  loss = 4.8    第一遍做题，啥也不会
Epoch 3:  loss = 0.14   第三遍，有点印象了
Epoch 10: loss = 0.0003 第十遍，滚瓜烂熟

但注意：刷太多遍可能"背答案"（过拟合），遇到新题就不会了。
```

**梯度和反向传播：**

```
前向传播（做题）:
  输入 → 模型计算 → 输出预测 → 和正确答案比 → 得到 loss

反向传播（对答案）:
  loss → 对每个参数计算梯度 → 更新参数
         （该往大调还是小调、调多少）

学习率（步长）:
  学习率 = 0.01    每步跨大步 → 学得快但容易跳过最优解
  学习率 = 0.00001 每步跨小步 → 学得慢但更精细

SFT 通常用很小的学习率（1e-5 ~ 5e-5），
因为预训练已经学到很多能力了，不想改太多把原有能力搞坏。
```

## 六、动手：从零搭建一个 Transformer

理论看完了，最好的理解方式是自己写一个。`transformer_demo.py` 用纯 PyTorch 实现了一个完整的 Transformer 语言模型（不依赖任何预训练模型），训练它学会几句中文。

### 运行

```bash
cd 01-基础概念
python transformer_demo.py
```

### 模型结构

整个模型只有 **0.21M 参数**（21 万），但结构和 GPT、Qwen 完全一致：

```
字符 → Token Embedding → Positional Encoding → Transformer Block × 4 → LayerNorm → 输出层 → 预测下一个字
```

### 七个组件逐个拆解

#### 1. Token Embedding

把每个字变成一个向量。就是一张查找表，每个字对应一行。

```
"天" → id=15 → [0.12, -0.34, 0.56, ...]  （d_model 维向量）
```

#### 2. Positional Encoding（位置编码）

Transformer 没有循环结构，它看到的是一堆向量的**集合**。如果不加位置信息：

```
"我爱你" 和 "你爱我" → 对模型来说完全一样！
```

位置编码用 sin/cos 函数给每个位置生成独特的编码，加到 Embedding 上。

#### 3. Self-Attention（自注意力）

**Transformer 的灵魂。** 每个字生成三个向量：

```
Q (Query)  —— "我在找什么？"
K (Key)    —— "我有什么特征？"
V (Value)  —— "我的实际内容"
```

用 Q 和所有 K 做**点积**，得到注意力权重，再加权求和 V：

```
Attention(Q, K, V) = softmax(Q · K^T / √d) · V
```

举例：处理"天问一号"中的"号"时，它的 Q 会和每个字的 K 做点积，点积越大说明越相关，注意力权重就越高。

#### 4. Multi-Head Attention（多头注意力）

一个头只能学一种关注模式。多个头各学各的，最后拼起来：

```
Head 1: 关注相邻的字（语法关系）
Head 2: 关注语义相关的字
Head 3: 关注结构和标点
Head 4: 关注长距离依赖
→ 拼接后通过线性层融合
```

#### 5. Feed-Forward Network（前馈网络）

Attention 负责"看其他位置的信息"，FFN 负责"消化这些信息"：

```
x → Linear(d_model → 4×d_model) → ReLU → Linear(4×d_model → d_model)
```

先升维再降维，中间用非线性激活，增加模型的表达能力。

#### 6. Transformer Block（残差 + LayerNorm）

把 Attention 和 FFN 包起来，加上残差连接和 LayerNorm：

```
输入 ──────────────────────────┐
  ↓                            │
LayerNorm → Multi-Head Attn    │
  ↓                            │
相加（残差连接）←──────────────┘
  │
  ├─────────────────────────┐
  ↓                         │
LayerNorm → FFN             │
  ↓                         │
相加（残差连接）←───────────┘
  ↓
输出
```

残差连接让梯度能直接流过去，不然层数一多就训不动了。

#### 7. Causal Mask（因果掩码）

语言模型生成时是一个字一个字往后写的，所以训练时每个位置**不能偷看未来的字**：

```
处理 "天" 时: 只能看 [天]
处理 "问" 时: 只能看 [天, 问]
处理 "一" 时: 只能看 [天, 问, 一]
处理 "号" 时: 只能看 [天, 问, 一, 号]
```

实现方式是在 attention score 上加一个**下三角矩阵**的 mask，把右上角设为 -∞，softmax 后就变成 0。

Epoch  50/100  Loss: 0.82    ← 开始学会了

生成测试:

  '北斗' → 北斗导航覆盖全球。神舟飞船载人航天。

```

### 注意力可视化

demo 会打印注意力权重矩阵，可以直观看到每个字在"关注"谁：

```
           天     问     一     号     成     功     着     陆     火     星
  天  │ 1.00  0.00  0.00  0.00  ...                        ← 第一个字只能看自己
  问  │ 0.84  0.16  0.00  0.00  ...                        ← 主要关注"天"
  一  │ 0.00  1.00  0.00  0.00  ...                        ← 关注"问"

### 和真实大模型的区别

| | 我们的 demo | Qwen2.5-0.5B | GPT-4 |
|------|-----------|-------------|-------|
| 参数量 | 0.21M | 500M | ~1,800,000M |
| 层数 | 4 | 24 | ~120 |
| 注意力头数 | 4 | 14 | ~96 |
| 隐藏维度 | 64 | 896 | ~12,288 |
| 词表大小 | 54（字符） | 151,936（子词） | ~100,000 |
| 训练数据 | 5 句话 | 几万亿 token | 更多 |

**结构完全一样，只是规模不同。** 理解了这个 demo，你就理解了所有大模型的骨架。

## 总结

```
大模型 = 一大堆参数（数字），通过预测下一个字来"说话"

内部结构:
  Embedding（查字典）→ Transformer x N（理解上下文）→ 输出层（预测下一个字）

Attention = 每个字看所有其他字，判断谁最重要
  Query（我在找什么）、Key（我有什么）、Value（我的内容）
  Multi-Head = 多角度理解

Tokenizer = 把文字切成 token，模型处理的最小单位

SFT = 用标注数据微调，只对回答部分算 loss（label=-100 跳过指令部分）
  预训练教说话 → SFT 教听话
```
