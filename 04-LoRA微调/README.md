# LoRA 微调：用 1% 的参数学会新技能

> 核心思想：冻住大模型，只训练旁边贴的"便利贴"。

---

## 一、为什么需要 LoRA？

全量微调（Full Fine-Tuning）虽然效果好，但有三个大问题：

| 问题 | 举例 |
|------|------|
| **灾难性遗忘** | 5亿参数全调，容易把预训练学到的通用能力覆盖掉 |
| **显存爆炸** | 7B 模型全量微调需要 60GB+ 显存，普通 GPU 根本跑不了 |
| **存储成本高** | 每个下游任务都要保存一份完整模型（7B = 14GB 磁盘） |

**LoRA 一次性解决了这三个问题：**

- 只训练 **不到 1%** 的参数 → 不容易遗忘原始能力
- 显存占用降低到 **1/4 ~ 1/10** → 消费级 GPU 也能跑
- 每个任务只保存一个小小的 adapter 文件（几十 MB）→ 存储成本极低

## 二、LoRA 原理

### 2.1 核心思想：低秩分解（Low-Rank Adaptation）

LoRA 的做法非常巧妙：

1. **冻结**原模型的所有参数（一个都不动）
2. 在原始权重矩阵旁边，**新增两个小矩阵 A 和 B**
3. 前向传播时：`Y = W × X + A × B × X`
4. 反向传播时：只更新 A 和 B，W 保持不变

### 2.2 直觉类比

> 原模型就像一本**教科书**（不允许修改），LoRA 就是在旁边贴的**便利贴**。
> 
> 学新知识时，你不需要重写整本书，只需要在关键页面贴上便利贴就行了。
> 便利贴很小、很轻，但能让你快速找到新的知识。

### 2.3 数学上有多省？

```
原始计算:  Y = W × X
  W 的大小: 4096 × 4096 = 16,777,216 个参数（16M）

LoRA 计算: Y = W × X + A × B × X
  A 的大小: 4096 × 8  =  32,768 个参数
  B 的大小: 8 × 4096  =  32,768 个参数
  合计:                    65,536 个参数（65K）

节省比例: 65K / 16M = 0.4%，只需要原来的 0.4% 参数！
```

其中 `r=8` 就是所谓的**秩（rank）**，它决定了便利贴的大小。

## 三、关键超参数

### 3.1 rank（r）—— 便利贴的大小

| rank 值 | 特点 | 适用场景 |
|---------|------|---------|
| r=4 | 参数最少，训练最快，但容量有限 | 简单任务、数据量少 |
| **r=8** | **默认推荐，平衡性最好** | **大多数场景** |
| r=16 | 容量更大，训练稍慢 | 复杂任务、数据量大 |
| r=32~64 | 接近全量微调的容量 | 特别复杂的任务 |

> 类比：rank 就是便利贴的大小。r=4 是小号便利贴，r=64 是整页便利贴。
> 大多数时候，**一个中号便利贴（r=8）就够了**。

### 3.2 alpha（α）—— 学习强度

- **缩放公式**：`lora_scaling = α / r`
- 这个缩放系数控制 LoRA 对输出的影响力度
- **经验法则**：`α = 2 × r`（比如 r=8 时设 α=16）
- α 太小 → LoRA 几乎不起作用
- α 太大 → 训练不稳定

### 3.3 target_modules —— 在哪些层贴便利贴

不是每一层都需要加 LoRA，常见的配置：

| 模型 | 推荐 target_modules | 说明 |
|------|---------------------|------|
| Qwen2.5 | `q_proj, v_proj` | 注意力层的 Q 和 V 投影，性价比最高 |
| Qwen2.5 | `q_proj, k_proj, v_proj, o_proj` | 注意力层全加，效果更好 |
| LLaMA | `q_proj, v_proj` | 和 Qwen 类似的架构 |
| 全部线性层 | `all-linear` | PEFT 支持的快捷写法，效果最好但参数更多 |

> 一般只对注意力层加 LoRA，不对 embedding 层和 LM head 加。

### 3.4 lora_dropout —— 防止过拟合

- 通常设置为 **0.05 ~ 0.1**
- 数据量小时可以适当调大（0.1）
- 数据量大时可以设小一些（0.05）或者设为 0

## 四、QLoRA：量化 + LoRA，显存再砍一刀

QLoRA 的思路非常简单：

1. 先把基座模型**量化到 4-bit**（从 FP16 的 14GB 缩到约 4GB）
2. 在量化后的模型上**再叠加 LoRA**
3. 训练时只更新 LoRA 的参数，基座模型保持 4-bit 不动

关键技术：
- **NF4 量化**：专门为正态分布的权重设计的 4-bit 量化格式
- **双重量化（Double Quantization）**：连量化的缩放因子都再量化一次，进一步省内存

### 方法对比

| 方法 | 7B 模型显存 | 训练效果 | 能跑的 GPU |
|------|-----------|---------|-----------|
| 全量微调 | 60GB+ | 最好 | A100 80GB |
| LoRA | ~16GB | 接近全量微调 | V100 32GB / A100 |
| **QLoRA** | **~6GB** | **略低于 LoRA** | **RTX 3090 24GB / RTX 4090** |

> QLoRA 让普通玩家也能在消费级显卡上微调 7B 甚至 13B 模型，是目前最流行的微调方式。

## 五、LoRA vs 全量微调 详细对比

| 维度 | 全量微调 | LoRA |
|------|---------|------|
| 可训练参数 | 100% | 0.1% ~ 1% |
| 显存占用 | 非常高（需要存所有梯度） | 低（只存 LoRA 梯度） |
| 训练速度 | 慢 | 快 2~3 倍 |
| 灾难性遗忘 | 容易发生 | 不容易（原参数冻结） |
| 存储成本 | 每任务一份完整模型 | 每任务一个小 adapter（几十 MB） |
| 多任务切换 | 需要加载不同模型 | 换一个 adapter 就行 |
| 效果上限 | 最高 | 接近全量微调（通常差 1~2%） |
| 适用场景 | 数据充足 + 充足算力 | 数据有限 / 算力有限 / 快速验证 |

**结论：对于大多数实际场景，LoRA 是性价比最高的选择。**

## 六、常见问题 FAQ

**Q: rank 该选多大？**
> A: 先试 `r=8`，这是最通用的选择。如果欠拟合就加到 16 或 32，如果过拟合就减到 4。

**Q: 要不要对 embedding 层加 LoRA？**
> A: 通常**不需要**。embedding 层参数量大，加 LoRA 会显著增加可训练参数量，但效果提升不明显。

**Q: 多个 LoRA adapter 能合并吗？**
> A: 可以。训练完成后，可以把 LoRA 的权重 merge 回原模型：`model = model.merge_and_unload()`。合并后就是一个普通模型，不再需要 PEFT 库。

**Q: LoRA 模型怎么部署？**
> A: 两种方式：
> 1. **合并部署**：merge 回原模型后，和普通模型一样部署
> 2. **分离部署**：用 PEFT 加载基座模型 + adapter，更灵活但需要 PEFT 库

**Q: LoRA 和 Prompt Tuning、P-Tuning 有什么区别？**
> A: 都属于参数高效微调（PEFT），但 LoRA 直接作用于模型权重矩阵，效果通常最好。Prompt Tuning 只在输入前加可训练的虚拟 token，更轻量但效果稍差。
